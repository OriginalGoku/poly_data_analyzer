# Lessons Learned

- Plotly `add_vline` with `annotation_text` silently fails on subplots with datetime x-axes. Use `add_shape` + `add_annotation` directly instead.
- Whale side attribution should use taker trades only -- maker trades are passive and don't indicate directional intent. A wallet that provides liquidity on both sides would appear "Mixed" incorrectly if maker trades were included.
- Wallet classification thresholds (maker % for Market Maker, minimum trade count) need to be configurable -- different markets have different liquidity profiles. Externalizing to `chart_settings.json` avoids hardcoding assumptions.
- Whale leaderboards need separate inclusion rules for aggressors vs liquidity providers -- qualifying as a whale on total volume is not enough to belong in a taker-only leaderboard.
- Cross-game analytics should come from a cached game-level table, not ad hoc scans inside UI callbacks. Otherwise sport filters and regime bands turn into repeated archive walks.
- For regime analysis, raw checkpoint opening prices can be too noisy. Re-anchoring "open" to the first pregame trade after cumulative volume reaches a minimum threshold produces a more defensible market-open proxy.
- Per-event sensitivity should be cached by game once computed. Replaying the same game in the dashboard should reload `cache/{date}/{match_id}_sensitivity.json` instead of recomputing VWAP windows on every callback.
- Backtest module organization works well when config is immutable (frozen dataclass) and orchestration is separate from single-game logic. A grid runner that loads games by date range and delegates to single-game functions keeps business logic decoupled from I/O.
- Exit strategy logic benefits from being factored into small, testable functions (dip detection, settlement resolution, PnL computation) rather than one monolithic function. This allows building baseline strategies (buy-at-open, buy-at-tipoff) that reuse the same exit and PnL machinery.
- Backtest result aggregation should preserve both per-game trades and aggregated statistics. Per-game records enable drill-down analysis; aggregations (return %, win rate, Sharpe) are needed for cross-strategy comparison.
- Fee models should be baked into the config and consistently applied. Taker fees (0.2%) significantly impact profitability; offering a "maker" model (0%) for comparison shows sensitivity to fees even if it's unrealistic for early adopters.
- **Data analytical work that processes many records (backtests, aggregations, exports) must provide real-time terminal progress feedback (tqdm, logging).** Without progress bars, users stare at blank screens wondering if the process hung or is genuinely working. Progress output (game count, config count, completion %) costs minimal overhead and dramatically improves user experience during long-running analyses.
- Exit type implementations with multiple compounding bugs (off-by-one in time anchor + wrong duration constant + skipped sport branch) are harder to fix than to remove if no production users exist. Delete cleanly rather than patch incrementally.
- Post-game trade data often contains settlement-price spikes that are not real price movements. Exit searches must be bounded to `< game_end` to prevent these artifacts from triggering spurious exits and inflating returns.
- Aggregated gross ROI should be derived from `gross_pnl / (entry_price * 100)` directly, not from the net `roi_pct` column, or fees will be double-counted in the mean metric.
- For pluggable components (filters/triggers/exits), keep the param schema next to the component itself as a `PARAM_SCHEMA` list and expose it through a parallel `*_SCHEMAS` registry. Builder UIs render inputs from the schema without duplicating it; adding a new component costs only the new file plus a one-line registration.
- When a per-game bulk pipeline reads each raw file twice (e.g. analytics scan + detail load), favor a streaming helper that yields `(base_record, get_game)` over a retain-in-dict approach. Retaining ~1000 decompressed games can blow past available RAM (~7 GB for NBA); streaming caps memory at one game and still wins the I/O.
- On-disk per-game caches that back user-visible pages need `input_fingerprint` (`mtime_ns` + size of raw files) in addition to `schema_version` + `settings_hash`. Other per-game caches can rely on the implicit "raw data immutable once collected" contract, but the user-visible perf cache must self-invalidate when raw data is re-collected.
- Dash callbacks that "populate downstream options" must not depend on Inputs that the user actually wants to leave stable. Toggling price-quality silently re-fired `populate_dates`, which reset start/end and retriggered the heavy `update_analysis`. Bind populate-once callbacks to a hidden `dcc.Store` mount trigger instead.
- `lru_cache` keys that include date-range parameters create cache thrashing whenever the user picks a new window. Drop the range from the cache key and apply the date filter in the caller — but place the filter BEFORE the quantile-source assignment, or quantile bands silently flip from window-local to all-history.
- Engine loops that only tick `PositionManager` *before* opening positions silently break scanner-driven exits combined with `sequential` lock: once locked, the trigger loop early-exits and the scanner never sees later trades, so positions only ever resolve via `force_close`. Always run a final `pm.tick(ctx, game_end)` before `force_close_all` so exits get the full tape. Pair this with an explicit `__call__` alias on any scanner class meant to be invoked as a function — duck-typed callable contracts are easy to break with class-based exit refactors.
- One knob can carry two comparators if the units happen to coincide visually. `pregame_min_cum_vol` is share-count cumulative `size` inside `_filter_by_min_cum_vol` (meaningful-open anchor) but USDC notional inside the main-dashboard game-picker gate (`min_pregame_notional` vs `manifest.volume_stats.pre_game_notional_usdc`). Document the dual semantics where the knob is defined — same field, different scales.
- Adding columns to a persisted base-records frame is a silent on-disk schema break for users with stale pickles. Either bump a schema version in the filename / sidecar, or surface a "delete `cache/_base_records/` once" note in README and the plan's risk flags. Don't rely on users to read the changelog.
- A `manifest.status == "collected"` game with implausibly low `volume_stats` (e.g. `trade_count: 2`, `total_notional_usdc: $5`) is NOT necessarily a truncated download. `poly-data-downloader`'s `rehydrate` only repairs files where `history_truncated: True` or `source: data_api` near the 4000-row cap; Goldsky-sourced files with sparse but authoritative data look identical to genuinely thin markets. Re-running `rehydrate` after flipping `history_truncated` returns the same row count, confirming the upstream Goldsky dataset itself only has those rows. Possible root causes upstream: (a) scan-time slug→condition_id mismatch onto an orphan/duplicate market, (b) genuinely stillborn market. The analyzer dashboard treats these the same way — hide via the `pregame_min_cum_vol` gate, badge near-misses via `data_warning_min_pregame_vol`.
